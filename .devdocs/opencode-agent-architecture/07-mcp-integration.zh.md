# 07 MCP Integration

MCP is the standard protocol layer for connecting external capabilities to an Agent. The Agent system should not treat MCP tools as special branches; instead, it should adapt them into ordinary tool values generated by `Tool.make(...)` and then pass them through `tools.register(...)` into the unified tool registration, permission evaluation, execution, and event logging flow.

> Source consistency note: The opencode product already has MCP-related configurations, UI contexts, and tests, but in V2 runner parity, MCP/plugin tool materialization is still partial/follow-up. This chapter describes the target design for replicating the complete MCP module; for the current V2 tool core, MCP tools should ultimately become `Tool.make(...)` values and be registered via `tools.register({ mcp__server__tool: tool })`.

## Module Responsibilities

This module is responsible for:

- Reading MCP server configurations.
- Starting, connecting, reconnecting, and closing MCP clients.
- Discovering tools, resources, and prompts exposed by MCP servers.
- Converting MCP tools into internal `Tool.make(...)` tool values.
- Converting MCP resources/prompts into Context Sources or user-selectable contexts.
- Delegating calls to MCP servers during tool execution.

This module is NOT responsible for:

- Bypassing ToolRegistry.
- Bypassing PermissionService.
- Directly feeding all MCP capabilities to the model.

## MCP Server Configuration

```ts
type MCPServerConfig = {
  id: string
  name?: string
  enabled: boolean
  transport:
    | {
        type: "stdio"
        command: string[]
        cwd?: string
        env?: Record<string, string>
      }
    | {
        type: "http"
        url: string
        headers?: Record<string, string>
      }
  trust?: "local" | "workspace" | "remote"
  toolVisibility?: "hidden" | "manual" | "model"
  permissionDefault?: "allow" | "ask" | "deny"
}
```

Design highlights:

- `id` is the stable identity in permissions, namespaces, and event logs.
- `trust` affects the default permission policy but is not equivalent to automatic allowance.
- `toolVisibility = "model"` indicates it can enter the model-visible list of ToolRegistry, but is still subject to Agent and permission filtering.

## MCP Client Manager

```ts
interface MCPClientManager {
  startAll(configs: MCPServerConfig[]): Promise<void>
  start(serverID: string): Promise<MCPClientHandle>
  stop(serverID: string): Promise<void>
  get(serverID: string): Promise<MCPClientHandle>
  list(): Promise<MCPClientHandle[]>
  refresh(serverID: string): Promise<MCPDiscoverySnapshot>
}

type MCPClientHandle = {
  serverID: string
  state: "starting" | "ready" | "failed" | "stopped"
  client: MCPClient
  config: MCPServerConfig
  lastError?: string
}
```

Lifecycle:

```text
load config
  -> start enabled servers
  -> initialize protocol
  -> discover capabilities
  -> register tools/resources/prompts
  -> keep handle for execution
```

## Discovery Model

```ts
type MCPDiscoverySnapshot = {
  serverID: string
  tools: MCPTool[]
  resources: MCPResource[]
  prompts: MCPPrompt[]
  discoveredAt: number
}

type MCPTool = {
  name: string
  description?: string
  inputSchema: JSONSchema
}

type MCPResource = {
  uri: string
  name?: string
  description?: string
  mimeType?: string
}

type MCPPrompt = {
  name: string
  description?: string
  arguments?: JSONSchema
}
```

Discovery results should be cached to facilitate:

- Displaying MCP capabilities in the UI.
- Retaining the previous tool list when an MCP server is temporarily unavailable, though execution should still fail or reconnect.
- Recording a snapshot of visible MCP tools when generating Context Epoch.

## MCP Tool Conversion

```ts
function mcpToolToCoreTool(input: {
  server: MCPServerConfig
  tool: MCPTool
  clientManager: MCPClientManager
}): Tool.AnyTool {
  return Tool.make({
    description: input.tool.description ?? `MCP tool ${input.tool.name}`,
    input: schemaFromMCP(input.tool.inputSchema),
    output: MCPToolOutput,
    execute: (toolInput, context) =>
      Effect.gen(function* () {
        yield* permission.assert({
          sessionID: context.sessionID,
          agent: context.agent,
          source: { type: "tool", messageID: context.assistantMessageID, callID: context.toolCallID },
          action: `mcp.${input.server.id}.${input.tool.name}`,
          resources: [input.server.id],
          save: [input.server.id],
          metadata: { serverID: input.server.id, tool: input.tool.name },
        })
        const handle = yield* Effect.promise(() => input.clientManager.get(input.server.id))
        const result = yield* Effect.promise(() =>
          handle.client.callTool({
            name: input.tool.name,
            arguments: toolInput,
          }),
        )
        return normalizeMCPToolResult(result)
      }),
  })
}
```

Conversion rules:

- Add server namespace to registration name, e.g., `tools.register({ mcp__server__tool: tool })`.
- Preserve the original MCP tool name in metadata, restoring it during execution.
- MCP input schema goes into unified schema validation.
- MCP result is converted to the wrapped tool's typed output, then projected through `toModelOutput(...)` and unified ToolRegistry settlement/bounding.
- MCP permission facts should mark the call as external; if the server is remote, include network access facts for default policy decisions.

## MCP Tool Result

MCP return content may include text, images, resource links, errors, etc. Unified internally as:

```ts
type NormalizedMCPToolResult = {
  output: string
  data?: unknown
  media?: FileAttachment[]
  metadata: {
    serverID: string
    originalContentTypes: string[]
  }
}
```

Conversion strategy:

- Merge text content into `output`.
- Store image content in blob store and as `media`.
- Resource links can be converted to attachment references or context resource references.
- MCP errors are converted to `ToolError` to prevent provider-specific exceptions from leaking to the Runner.

## MCP Resource Integration

Resources are not necessarily tools. They are better suited as selectable contexts.

```ts
interface MCPResourceService {
  list(input: { serverID?: string }): Promise<MCPResource[]>
  read(input: {
    serverID: string
    uri: string
    signal: AbortSignal
  }): Promise<MCPResourceContent>
}

type MCPResourceContent = {
  uri: string
  mime: string
  text?: string
  blob?: FileAttachment
}
```

Ways to bring resources into context:

1. User selects a resource in the UI to attach to the prompt.
2. Agent configuration declares certain resources to automatically enter Context Source.
3. MCP tool returns a resource link, which the model or user decides whether to read.

Automatic injection should be done cautiously, as resources may be large or sensitive.

## MCP Prompt Integration

MCP prompts can serve as templates or commands; they are not recommended to automatically enter the system prompt.

```ts
interface MCPPromptService {
  list(input: { serverID?: string }): Promise<MCPPrompt[]>
  render(input: {
    serverID: string
    promptName: string
    arguments: unknown
  }): Promise<SessionMessage[]>
}
```

Usage approach:

- List MCP prompts in the UI command palette.
- User selects and fills in parameters.
- Render as user message or system context, then follow normal prompt admission.

## Permission Design

MCP permissions need to distinguish three things:

- Starting an MCP server.
- Exposing an MCP tool to the model.
- Executing a particular MCP tool call.

Permissions for starting a server:

- stdio server starts a local process, which is a `process.exec` risk.
- HTTP server accesses the network, which is a `network.access` risk.
- If the configuration comes from a user-explicit file, it can be considered already allowed to start, but still needs logging.

Permissions for executing a tool:

```ts
type MCPPermissionFacts = {
  serverID: string
  serverTrust: "local" | "workspace" | "remote"
  toolName: string
  arguments: unknown
  transportType: "stdio" | "http"
}
```

Recommended default policies:

- local/workspace stdio server: default ask, user can allow per session/workspace.
- remote HTTP server: default ask, with host indication.
- Unknown or dynamic server: default deny or ask with high risk.

## Event Design

```ts
type MCPEventType =
  | "mcp.server.started"
  | "mcp.server.failed"
  | "mcp.server.stopped"
  | "mcp.discovery.completed"
```

Tool execution itself still uses `session.next.tool.called`, `session.next.tool.success`, and `session.next.tool.failed`; no separate MCP tool events are needed. MCP metadata can be placed in the tool payload or settlement result.

## Reconnection and Failure

Strategy:

- Server startup failure should not block the entire Agent startup, only disable that server's tools.
- If connection is lost during execution, attempt one reconnect.
- After reconnection, re-run discovery and update ToolRegistry.
- If tool schemas change after discovery, the tool spec already exposed to the model in the current round remains unchanged; update in the next round.

## Implementation Steps

1. Define `MCPServerConfig`.
2. Implement stdio/http client manager.
3. Implement discovery snapshot.
4. Register MCP tools with ToolRegistry.
5. Implement MCP tool execute delegation.
6. Integrate MCP facts into PermissionService.
7. Implement resource list/read.
8. Implement prompt list/render.
9. Add reconnection, failure degradation, and events.

## Acceptance Criteria

- After startup, MCP tools can be discovered and registered with namespaced tool names.
- MCP tools do not override built-in tools.
- Unauthorized MCP tool calls enter permission pending state.
- MCP tool results can be converted to ordinary tool results.
- After an MCP server disconnects, execution failure is readable, and Runner can continue.
- Resources can be selected by the user to enter the prompt, rather than being automatically injected entirely.

## Common Pitfalls

- Exposing all tools to the model immediately after the user configures an MCP server.
- MCP tools bypassing the unified permission system.
- MCP tool names not being namespaced, causing conflicts with built-in tools.
- Automatic injection of resource content leading to context overflow or leaks.
- Not refreshing the registry after MCP server schema changes.
